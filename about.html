Methodology

How sensing digests are produced and governed.

What this site is

Social Sensing is a personal sensing log tracking what is being said about Singapore tech, telco, and media policy — by whom, where, and how framed. Digests are produced as situational awareness only. They contain no strategy, no recommendations, no predictions, no official position of any organisation. Views and interpretations expressed in the digests are the author's, drawn from publicly observable social commentary and news reporting.

The site is publicly accessible. There is no access gate. Anyone with the URL can read every digest in the archive.

Schedule

A digest is produced once each weekday morning at 08:00 SGT, covering the trailing 7-day window from the run time. The analyst may also fire manual runs on demand with a different cadence (e.g. a 24-hour catch-up window after a major event). The cadence and window for each digest are named in the page header and the footer methodology line.

The scheduled run only fires while the controlling desktop app is open. If closed at 08:00 SGT, the run lands on the next app launch. The system is best-effort, not guaranteed daily.

Data sources

Data-source mode is hybrid by default. On every run the tool reads both available sources:

  1. Analyst-supplied source. Files dropped into the inputs folder (CSV, TXT, PDF, DOCX, RTFD, or WEBARCHIVE) are read and filtered to the trailing window. This is typically a social-listening export carrying handles, engagement metrics, and content from platforms that the live web scan cannot reach (Reddit, behind-login content).
  2. Supplementary live web scan. The tool runs a search against an accessible subset of a hand-maintained whitelist of Singapore mainstream news, forums, government sites, and selected tech press. Results are filtered to items published within the window.

The whitelist has a known structural blind spot: the web crawler used for live scanning is blocked from several high-value Singapore domains (the major mainstream titles and Reddit). Content from those domains can only enter a digest via the analyst-supplied export. The blocked-domain list is named in the footer of every digest so a reader can see what was structurally absent from the scan.

How digests are produced

Each digest is generated with AI assistance. The tool reads the whitelist and the scan-terms configuration, runs the hybrid ingest above, clusters signal under the seven regulated domains (below), and writes the digest as an HTML page. The archive index is regenerated on every run. Each run is logged to an audit trail.

After each run, the new digest is committed and pushed to the GitHub repository that backs this site. GitHub Pages serves the contents publicly within minutes. There is no per-run human review before publication. If a digest is wrong, misleading, or based on a failed run, the analyst retracts it by renaming the file — the retracted file disappears from the archive on the next push but remains on disk for audit.

Topic taxonomy

Clusters map to one of seven regulated domains in Singapore tech and media:

  • AI and digital technology
  • Telecommunications
  • Online safety and scams
  • Broadcast media and content
  • Digital identity and data protection
  • Postal and digital services
  • Broadband and connectivity

Anything outside these seven is out of scope and is dropped silently.

Editorial standards

  • British English. Plain English. Bullets, not paragraphs.
  • Sentiment is qualitative only. No percentages, no share-of-voice, no volume counts.
  • One verbatim quote per cluster maximum, under 15 words, always attributed.
  • Sources are linked to the original posts where the underlying export carries usable URLs. Closed or unlinkable channels are noted as such with platform-level counts — never fabricated.
  • Anomaly flags only when activity or tone is materially different from the prior baseline.

AI Governance — MGF for Agentic AI

This sensing tool is mapped against the Singapore Infocomm Media Development Authority's Model AI Governance Framework for Agentic AI (updated 20 May 2026). The framework is voluntary but treated here as the operational baseline.

Dimension 1 · Assess and bound the risks

  • Strong fit
  • External read access is bounded to a hand-maintained source whitelist. Nothing outside the whitelist is fetched.
  • External write access is bounded to a single GitHub repository under the analyst's control. The tool publishes to that repository only — no email, no messaging, no API calls to third-party services, no payments.
  • Content scope is sensing only — no decisions, no transactions, no actions on behalf of any organisation or individual.

Dimension 2 · Meaningful human accountability

  • Gap
  • Each digest is generated autonomously and pushed to the public site without per-run human review. There is no in-flight checkpoint — the digest goes from generation to publicly readable in one continuous pipeline.
  • The mitigations that exist are after-the-fact only: the runs.log audit trail supports retrospective review; the analyst can retract a digest at any time by renaming the file and pushing the rename, which removes the digest from the public archive on the next push; a periodic spot-check ritual surfaces patterns of error or drift.
  • This Gap rating is the honest accounting of an architectural trade-off. The system was designed to remove per-run friction. If the cost becomes unacceptable, three paths back toward Partial or Strong fit exist: (a) push to a staging branch and require a manual merge to main before publication; (b) add a post-run notification with an explicit publish/reject action; (c) introduce a pre-publish queue that requires analyst approval before the archive lists a new digest.

Dimension 3 · Technical controls and processes

  • Strong fit
  • Folder isolation. The tool operates inside a single root directory and writes only to subfolders within it.
  • Whitelist enforcement on every external fetch. Attempts to non-whitelisted domains are dropped and logged.
  • Pre-save quality checks on every digest including URL validity, length governance, sentiment-row consistency, footer inputs accuracy, and structural HTML validation.
  • The archive index is backed up before regeneration. The run log is append-only. Failed regenerations restore from backup.

Dimension 4 · End-user responsibility

  • Strong fit
  • This page is the disclosure surface. The site header on every digest links here, so any reader can see how a digest was produced before reading the content.
  • Public accessibility is disclosed upfront. The site has no access gate. Anyone with the URL can read every digest. This page (and this paragraph) is the canonical disclosure that the system has no per-run human review before publication.
  • End users are anyone who finds the URL. Every digest tells the reader the same things: it was generated with AI assistance; the data sources read for that run are named in the footer; sentiment markers are a qualitative read, not a measurement; views in the digest are the author's, drawn from publicly observable commentary.
  • Agent boundaries are disclosed explicitly. The tool reads only files inside the inputs folder or domains listed in the source whitelist. It writes only HTML files inside the site folder. It takes no other action.
  • Data-source mode is hybrid. When an analyst drops a supported source file into the inputs folder, the tool reads that file alongside the live scan. When no file is present, the tool runs the live scan alone. The mode that ran is named in the footer of every digest so a reader can judge data fidelity at a glance.
  • Override and retraction. Every digest is a plain HTML file. The analyst can edit, redact, or retract any digest by renaming it; the retraction removes the digest from the public archive on the next push, while the original stays on disk for audit. Corrections and scope changes route to the analyst via the contact path below.
  • Auditability. Each run is recorded in the run log with ISO timestamp, mode, source filename, extracted record count where relevant, tile count, check-pass summary, and git push status. The trail supports spot-check review at any time.
